* Summary

A user-authored skill that captures *what's installed, what's banned,
and where things live* on the user's machine. Loaded whenever the
agent is about to install software, build something, invoke an editor
/ formatter / LSP, or assume a path on disk. Part of the user-context
skill suite governed by [[file:suite-conventions.org][suite-conventions.org]].

* Motivation

Tool choices are the most visible class of environment-wrong agent
output (=nix profile install=, =apt=, =brew=, global =npm install -g=,
wrong build runner, wrong formatter). =my-tools= is the trigger window
for any decision about "which tool, where it lives, how to invoke it".

Why splitting this out (rather than rolling it into one omnibus skill)
is justified is documented in [[file:research.org][research.org]]; the cross-cutting
naming and description conventions live in [[file:suite-conventions.org][suite-conventions.org]].

This skill folds in the previously-proposed =my-editor= and =my-paths=
categories. Editor / LSP / formatter is "what's installed and how do
I invoke it"; "where do my repos live" co-occurs with install/build
decisions. Splitting them back out would put them in direct trigger
competition with =my-tools=, which is the documented #1 skill failure
mode.

* Proposal

** Trigger condition

Decisions about:

- Installing software (which package manager, which is banned)
- Picking a build tool / runner
- Invoking an editor, formatter, LSP, or linter
- Relying on a stable local path (where repos live, where scratch
  files go)

** Trigger keywords

(From the cross-suite matrix in [[file:suite-conventions.org][suite-conventions.org]].)

=install=, =add=, =fetch=, =build=, =editor=, =formatter=, =LSP=,
=repos=

* Design Details

** Future-extension hook

If a user's editor-tooling content grows large (multiple LSP configs,
formatter chains, plugin ecosystems), they can split it out into a
local =my-editor= skill (user extension; not a v0.1 ship). The suite
convention does not need to change to allow this.

** Skill-specific open questions

1. *Section count.* Six sections is on the upper bound of what stays
   readable. If real templates show users routinely deleting two or
   more, consider trimming the default.
2. *Editor / LSP placement.* Currently inline; some users might
   prefer it adjacent to =my-style= (since editor configures
   formatters that shape output). Lean keep-here on trigger-condition
   grounds.
