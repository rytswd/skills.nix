* Summary

A user-authored skill that captures the user's git, code-review, and
deployment workflow on this machine. Loaded before any
version-control action, PR / review interaction, or deploy. Part of
the user-context skill suite governed by [[file:suite-conventions.org][suite-conventions.org]].

* Motivation

Workflow preferences (rebase vs merge, force-push policy, who
reviews when, deploy mechanics) are user- and team-global and almost
always violated by default agent behaviour. =my-workflow= is the
trigger window for "about to do something git-adjacent or
deploy-adjacent".

Why splitting this out (rather than rolling it into one omnibus skill)
is justified in [[file:research.org][research.org]]; cross-cutting conventions live in
[[file:suite-conventions.org][suite-conventions.org]].

* Proposal

** Trigger condition

About to perform:

- A git operation (commit, push, pull, rebase, merge, force-push,
  cherry-pick, stash)
- Opening, reviewing, or merging a PR / MR
- Releasing or deploying

Note the partition with =my-style=: =my-workflow= covers *how* a
change is handled; =my-style= covers *what* the change's text says.

** Trigger keywords

(From the cross-suite matrix in [[file:suite-conventions.org][suite-conventions.org]].)

=git=, =PR=, =merge=, =deploy=, =review=, =rebase=, =force-push=

* Design Details

** Skill-specific open questions

1. *Deploy / release scope.* Some users have elaborate deploy
   pipelines that warrant their own skill (=my-deploy=). Lean keep
   here for v0.1; revisit if real templates outgrow the section.
2. *Code-review section.* Currently combined with PR opening; could
   be split if review preferences (granularity, response cadence)
   prove distinct from authoring preferences.
